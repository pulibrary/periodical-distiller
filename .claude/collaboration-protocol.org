#+TITLE: Three-Way Collaboration Protocol
#+SUBTITLE: Cliff + Claude Chat + Claude Code
#+DATE: 2026-02-26

* Overview

This document defines a protocol for collaboration between three
participants on a shared codebase, using org-mode files as the
primary coordination medium.

** Participants and Roles

- Cliff :: Decision-maker and domain expert.  Reviews all proposals,
  sets dispositions, writes Responses.  The only participant who
  commits to project direction.

- Claude Chat :: Research, architecture, and design.  Performs web
  research, explores tradeoffs, drafts design documents, reviews
  cross-project patterns.  Produces /Research/ and /Design/ notes
  that inform decisions.  Has broad context across projects but no
  direct access to the codebase.

- Claude Code :: Implementation and codebase analysis.  Lives in the
  repo.  Performs code review, grepping, refactoring, testing, and
  implementation.  Produces /Analysis/ notes grounded in actual code.
  Has deep context within one project but limited external research
  capability.

** Key Principle: Cliff Decides

Neither AI participant acts without Cliff's sign-off.  Claude Chat
proposes architectures; Claude Code proposes implementations; Cliff
approves, modifies, or rejects.  The org file is the decision record.

* The Org File as Shared Medium

** File Location and Name

The file is called =forum.org= — a place for the exchange of ideas
among participants.  It lives in the project repository root (or a
=docs/= directory).  Claude Code reads and writes it directly.  Cliff
edits it in Emacs.  Claude Chat receives it as an upload and produces
content that Cliff pastes or transfers.

The name signals intent: this is a deliberative space, not a task
queue.  Some items will become tasks; many will not.  The document's
value is as a project record, not just a checklist.

** Structure of an Item

The basic unit is an org headline — not necessarily a TODO.  Items
represent anything the project needs to track: questions, decisions,
research findings, bugs, features, or reference material.

The TODO keyword is applied only when an item represents *actionable
work with a definite completion state*.  Many items won't need it:
a research question that concludes "no action needed" is still a
valuable record.  A design decision is important to preserve even
after it's settled.

#+begin_example
,*** Short description of the item
:PROPERTIES:
:CUSTOM_ID: category/slug
:ASSIGNEES: Claude Code, Cliff
:SOURCE: chat/2026-02-26
:END:
Brief statement of the problem, question, or goal.  One paragraph,
enough for any participant to understand what this is about without
reading the children.

,**** Research
Notes from Claude Chat: background research, options analysis,
architectural reasoning, links to external resources.  Written in
prose, not as instructions.  This section answers "what should we
do and why?"

,**** Analysis
Notes from Claude Code: what the code actually does, grep results,
dependency analysis, impact assessment.  This section answers
"what does the code say and what would change?"

,**** Decision
Cliff's response.  States the chosen approach, any modifications
to the proposals, and the rationale.  Short is fine: "Go with
option 2" or "Defer until after v0.3."

,**** Implementation
Claude Code's record of what was done: files changed, tests added,
commit hash.  Written after the work is complete.
#+end_example

** When to Use TODO Keywords

Apply a TODO keyword when an item has become actionable — i.e., a
Decision has been made and there is concrete work to do.  The keyword
tracks the /work state/, not the item's overall lifecycle.

An item might pass through several phases without ever becoming a TODO:

| Phase            | Keyword  | Meaning                                      |
|------------------+----------+----------------------------------------------|
| Open question    | /none/   | Under discussion; no action yet               |
| Decided, no work | /none/   | Decision made; nothing to implement           |
| Actionable       | TODO     | Decision made; work to be done                |
| In progress      | ACTIVE   | Being worked on (optional; useful for Claude Code) |
| Complete         | DONE     | Work finished                                 |

Items that never become actionable simply accumulate Research,
Analysis, and Decision sections as a project record.  Org's property
search and sparse-tree features can surface them by =DISPOSITION=,
=CUSTOM_ID=, =ASSIGNEES=, or any other property — no keyword required.

Priority cookies (=[#A]=, =[#B]=, =[#C]=) are similarly optional and
apply only to items that have a TODO keyword.  They are useful for
triage but should not be applied to non-actionable items.

** Properties

| Property    | Purpose                                           | Values                                           |
|-------------+---------------------------------------------------+--------------------------------------------------|
| CUSTOM_ID   | Stable cross-reference slug                       | =category/short-slug=                            |
| ASSIGNEES   | Who is responsible for next action                 | =Cliff=, =Claude Code=, =Claude Chat=, or combo  |
| SOURCE      | Where the item originated                         | =chat/YYYY-MM-DD=, =review/YYYY-MM-DD=, =adhoc=  |
| DISPOSITION | Outcome of the Decision                           | =fix now=, =defer=, =document=, =wontfix=, =adopt= |
| COMMIT      | Commit hash after Implementation                  | Short SHA                                        |
| DEPENDS     | CUSTOM_ID of a blocking item                      | =category/other-slug=                            |

=DISPOSITION= is the primary state-tracking property.  It records
Cliff's decision and is set at the same time as the Decision section.
Org's property search (=C-c / m= for sparse tree, or custom agenda
commands) can filter on =DISPOSITION= to answer questions like "show
me everything deferred" or "what have we decided to adopt but not
yet implemented?"

** Section Conventions

Not every item needs all four sub-sections.  Use what's appropriate:

- Simple bug with no design questions :: Analysis + Decision (or just
  Implementation if trivial).  Likely gets a TODO keyword.
- Research question :: Research + Decision.  May or may not become
  actionable.
- Architectural decision :: Research + Analysis + Decision.  The
  decision is the point; implementation may follow or may not.
- Full feature :: All four sections.
- Reference / context :: Just the headline and body paragraph.  No
  sub-sections needed.  Useful for recording external constraints,
  stakeholder requirements, or background that multiple items refer to.

The key discipline: *Research and Analysis are proposals.  Decision is
the record of authority.  Implementation is the record of action.*

** Timestamps

Use an inactive timestamp at the start of every Decision section:

#+begin_example
,**** Decision
[2026-02-26] Go with option 2.  Defer the validator until after v0.3.
#+end_example

This is the one timestamp that matters: when Cliff made the call.
It enables filtering by date ("what did we decide this week?") and
tracing how the project's thinking evolved over time.

Timestamps on Research, Analysis, and Implementation sections are
welcome but not expected.  They are most useful when:
- An item accumulates multiple rounds of input over time
- The date of research matters (e.g., fast-moving external landscape)
- You want to correlate a section with a specific chat session

Do not timestamp for the sake of completeness.  The git log provides
a full audit trail; the forum's timestamps are for human orientation,
not archival rigour.

* Workflow Patterns

** Pattern 1: Research flows to Implementation

This is the Kraken GPU case.  Claude Chat does background research
and produces a Research section.  Cliff reviews and decides.  The
decision spawns one or more implementation TODOs assigned to Claude
Code.

#+begin_example
1. Claude Chat writes Research section with findings
2. Cliff writes Decision ("proceed with approach X")
3. Cliff (or Claude Chat) drafts child TODOs for implementation
4. Claude Code picks up TODOs, writes Analysis, implements, records
#+end_example

** Pattern 2: Code Review surfaces design questions

This is the current Princeton Distiller pattern.  Claude Code does
analysis and surfaces questions.  Some questions are purely
implementational (Claude Code + Cliff resolve them).  Others need
architectural input (Claude Chat weighs in with Research).

#+begin_example
1. Claude Code writes Analysis with open questions
2. Cliff triages: implementational → Decision directly
                  architectural → assigns to Claude Chat
3. Claude Chat writes Research section
4. Cliff writes Decision
5. Claude Code implements
#+end_example

** Pattern 3: Cliff brings a question

Cliff has a question or idea and doesn't yet know whether it needs
research, code analysis, or both.

#+begin_example
1. Cliff creates TODO with problem statement
2. Cliff assigns to Claude Chat, Claude Code, or both
3. Assigned participants fill in their sections
4. Cliff decides
#+end_example

* Handoff Conventions

** From Claude Chat to Claude Code

When Claude Chat's research produces actionable implementation tasks,
the Research section should end with a clearly marked block:

#+begin_example
Actionable for Claude Code:
- Specific thing to do, with enough context to act
- Another specific thing, referencing files or APIs by name
- Constraints or non-obvious requirements
#+end_example

This block is written /for Claude Code's consumption/.  It should be
concrete, reference specific files/functions/APIs where possible, and
state constraints.  It should not require Claude Code to understand the
full research narrative — just the conclusion.

** From Claude Code to Claude Chat

When Claude Code's analysis reveals questions that need broader
research, the Analysis section should end with:

#+begin_example
Questions for Claude Chat:
- What are the tradeoffs between X and Y in the broader ecosystem?
- Is there prior art for this pattern in similar projects?
- What does the spec/standard say about Z?
#+end_example

** From either AI to Cliff

Open questions requiring Cliff's judgment should always be explicit
and clearly separated from analysis:

#+begin_example
Open questions:
- Preferred approach: A (simpler) or B (more robust)?
- Priority relative to other work?
- Any constraints from stakeholders?
#+end_example

* What This Protocol Does Not Cover

- *Real-time collaboration*: The two Claude instances never interact
  directly.  Cliff is always the bridge.
- *Automatic syncing*: There is no tooling to push changes between
  Claude Chat and the repo.  Cliff copies content manually.
- *Access control*: Both AIs can in principle read and write any
  section.  The discipline is social, not technical.
- *Long-term memory*: Claude Chat's memory across sessions is
  imperfect.  Important context should be in the org file, not
  assumed to persist in chat history.

* Bootstrapping This Protocol

To adopt this protocol on a new project:

1. Create =forum.org= in the project repo
2. Add a reference to this protocol in the project's =CLAUDE.md=:
   : See forum.org for the project's deliberative record.  Follow
   : the collaboration protocol in docs/collaboration-protocol.org.
3. When starting a Claude Chat session about the project, upload
   =forum.org= so Claude Chat has current context
4. When items are created or updated in chat, Cliff transfers
   the content to the repo's =forum.org=
