 * One of the differences between the logical structMap in your data and what we normally see is that typically all of the divs tend to have ID's: (e.g. ID="DIVL15"). So typical in fact that Veridian expects this, and the lack of IDs cause errors in Veridian by default when importing this data. However, we see that having ID's on those divs is optional in the schema, so if you wish, we could customise Veridian to not expect those ID's. One way Veridian uses these IDs is during ingestion as a reference to determine where in the METS something is going wrong.

 * Another typical difference from Princetonian article-level METS/ALTO is that these Article div types don't have LABEL attributes. However, that should be fine, as you have provided the title metadata of the articles in MODS metadata which is associated with each article using DMDIDs, so we can determine what each article title is using those DMDIDs. There is code in Veridian to do this but for some reason it isn't working for this data so we would have to look further into why this is.

 * We also noted that you have type='INSIDE' page-level div types in this data, instead of the default 'PAGE' type, but we assume there may be a good reason for this, so for now we added that to the Papers of Princeton collection configuration of accepted page-level div types.

 * Probably the most important item to deal with next (in order to ingest the METS/ALTO as article-level as opposed to page-level) is ultimately associating the set of text blocks that make up each article with each article div in the Logical structMap. Currently you have:

If we definitively know that every page of this data will only ever contain one article, i.e. there are no pages which contain the end of one article and the start of another article, then we can infer that all of the ALTO files in each article-level directory contain all the text blocks that make up that article, and customise the relevant Veridian code to ingest this.

However, even this being the case, we believe it would be a lot clearer in the METS to at least list the ALTO FILEIDs for each article, inside each Article div type. This would also mean less complex customised code to ingest this data.
